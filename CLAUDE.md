# Project Instructions

## One-Click Runner Rule (CRITICAL)

This project has a **`run.sh`** bash script that non-technical users double-click to run the entire pipeline. Every time you make changes in a session that affect the pipeline, you MUST:

1. **Update `run.sh`** — Ensure it runs ALL pipeline steps in the correct sequential order. If you add new phases, scripts, notebooks, or dependencies, add them to `run.sh` so a single `bash run.sh` executes everything end-to-end.

2. **Update `requirements.txt`** — If you add any new Python dependency, add it here. `run.sh` installs from this file.

3. **Keep `run.sh` self-contained** — It must:
   - Create the `.venv` if missing
   - Install/update all dependencies
   - Run every pipeline step in order
   - Print clear output so a non-technical person knows what happened
   - Exit on first error (`set -euo pipefail`)

4. **Test `run.sh` after changes** — Run `bash run.sh` to verify the full pipeline still works end-to-end before finishing your session.

## Key Technical Decisions

- **DuckDB** for all data storage and SQL analytics (no pandas for transforms)
- **Immutability**: `raw_health` is never modified after creation; `clean_health` is a derived copy
- **Disease Category column** in the CSV is unreliable — replaced by hand-curated `DISEASE_CATEGORY_MAP` in `config.py`
- **Data quality flags** filter bad rows out of analytical views (`data_quality_flag = 'ok'`)
- **3 urbanization tier schemes** for sensitivity analysis (Question 4)

## Conventions

- Snake_case for all DuckDB column names
- Analytical view tables prefixed with `v_`
- All views enforce minimum observation counts (`SMALL_N_THRESHOLD = 30`)
- Pipeline halts on verification failure — fix the issue before proceeding
